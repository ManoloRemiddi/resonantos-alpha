## Overview

ResonantOS uses a 4-layer memory stack for context and continuity. Each layer serves a different purpose, persistence, and cost profile.

## The 4-Layer Memory Stack

| Layer | What | ~Tokens | Persistence | Delivery |
|-------|------|---------|-------------|----------|
| 1. MEMORY.md | Long-term curated knowledge | ~15K | Permanent, main session only | OpenClaw workspace injection |
| 2. RECENT-HEADERS.md | Last 20 memory log headers | ~5K | FIFO (count-based) | R-Awareness always-on docs |
| 3. LCM | Session history compression | Session | Automatic (context engine plugin) | LCM plugin (lossless-claw v0.5.2) |
| 4. RAG | Semantic vector search | On-demand | SQLite + Ollama embeddings | memory_search tool |

### Layer 1: MEMORY.md (~15K tokens)

**Purpose:** Curated long-term memory — decisions, rules, lessons, project state.
**Location:** `~/.openclaw/workspace/MEMORY.md`
**Loaded:** Every turn in main session (workspace context injection). NOT loaded in shared/group contexts (security).
**Maintenance:** Manual curation by agent. Periodically reviewed during heartbeats.

### Layer 2: RECENT-HEADERS.md (~5K tokens)

**Purpose:** Compressed session continuity — last 20 memory log headers (500-800 tokens each).
**Location:** `~/resonantos-augmentor/ssot/L1/RECENT-HEADERS.md`
**Loaded:** Every turn via R-Awareness V3 `appendSystemContext` (cacheable).
**Maintenance:** Auto-generated by `header-generation` cron (daily 06:00 Rome). 20-header FIFO.
**Source:** Memory logs in `memory/shared-log/MEMORY-LOG-YYYY-MM-DD[-suffix].md` (113+ files).

### Layer 3: LCM (Session-scoped)

**Purpose:** Lossless context management — compresses session history without losing information.
**Plugin:** lossless-claw v0.5.2, occupies the mutually exclusive `contextEngine` plugin slot.
**Config:** `incrementalMaxDepth: 3` (bounded to 3 condensation levels), `freshTailCount: 32`, `contextThreshold: 0.75`. ⚠️ Unlimited depth (-1) causes context_items bloat.
**Tools:** `lcm_grep`, `lcm_describe`, `lcm_expand`, `lcm_expand_query` for recall.

### Layer 4: RAG (On-demand)

**Purpose:** Semantic search across SSoT docs, memory logs, and research files.
**Embedding:** Ollama/nomic-embed-text (768d, local, ~4,810 chunks for main agent).
**Storage:** `~/.openclaw/memory/{agent}.sqlite` (per-agent).
**Access:** `memory_search` tool, Dashboard → Settings → Knowledge Base.

## Memory Log System

Memory logs are the raw input for Layers 1 and 2.

### Format: 3-Part DNA

| Part | Content |
|------|---------|
| Process Log | Strategic view — human input, AI analysis, decisions, failures, insights |
| Trilemma Diagnosis | Which failure pattern? F1 (Wrong Approach), F2 (Missing Data), F3 (Bad Prompt) |
| DNA Sequencing | Patterns extracted — Code, Violation, Correction, Impact, Recurrence Risk |

**Template:** `memory/shared-log/0000-PREAMBLE.md`
**SSoT:** `SSOT-L1-MEMORY-LOG.md`

### Memory Log Generation

| Cron | Schedule | Model | Purpose |
|------|----------|-------|---------|
| intraday-memory-log | Every 3h | MiniMax M2.7 | Live session → DNA-format log |
| daily-memory-log-dna | 04:30 Rome | MiniMax M2.7 | Nightly safety net |
| header-generation | 06:00 Rome | MiniMax-M2.7 | Rebuild RECENT-HEADERS.md from logs |
| Memory Archivist | 05:30 Rome | MiniMax-M2.7 | SSoT drift detection |

### Breadcrumbs

Real-time event capture: `memory/breadcrumbs.jsonl`. One-line JSON per significant event.
```json
{"ts":"ISO","event":"what","broke":"what went wrong","fcode":"F1|F2|F3|null","dna_tag":"TAG"}
```
Processed by intraday cron, cleared after memory log written.
